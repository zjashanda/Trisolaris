# 好太太晾衣机当前有效需求清单

更新时间：2026-04-22

## 1. 合并口径

本清单按以下优先级整理当前有效需求：

1. 基线文档：`电控词表协议--小好晾衣机语音词条协议-20260309.xlsx`
   - `电控协议`：命令词、发送协议、部分被动播报/品牌/上电逻辑
   - `协议播报`：MCU -> 语音模组的被动播报协议
   - `固件逻辑补充`：掉电保存、语音开关、播报开关逻辑
2. 覆盖文档：`好太太晾衣机需求迭代.md`
   - 对基线文档中的新增、删除、调换、特殊状态逻辑，以该文件为准
3. 行为补充：`语音关闭逻辑.png`
   - 用于明确“语音功能关闭后”的状态机细节
4. `fw-csk3022-xiaohao_clothes_airer-v1.0.10.zip`
   - 仅包含固件镜像，不提供额外需求规则

整理原则：

- Excel 与迭代文档冲突时，以 `好太太晾衣机需求迭代.md` 为准。
- Excel 内部若出现明显错位或字段缺失，本清单保留“当前最合理解释”，并在文末标记“待确认项”。
- 本清单面向“当前有效需求”，不再保留已被迭代文档删除的旧功能作为正式范围。

## 2. 项目定位与系统分工

- 产品是“语音模组 + 晾衣机 MCU”的协作方案。
- 语音模组的核心职责是：
  - 识别唤醒词/命令词
  - 将对应命令协议发送给 MCU
  - 按 MCU 回传协议执行被动播报
- 常规规则下，语音模组“识别命令后不直接播报业务结果”，业务结果播报以 MCU 回传协议为准。
- 少数全局状态命令是例外：
  - `打开静音 / 退出静音`
  - `打开播报 / 关闭播报`
  - `语音功能打开 / 语音功能关闭`
  这些命令带有本地状态切换属性，不能完全等 MCU 回传后再生效。

## 3. 通讯与基础运行规则

### 3.1 协议结构

- 常规通讯结构：`数据头（2Byte） + 数据方向（1Byte） + 数据内容（2Byte） + 累加和（1Byte） + 数据结束（1Byte）`
- 数据头：`0xA5FA`
- 数据结束：`0xFB`
- 语音模组 -> MCU 的数据方向：`0x7F`
- MCU -> 语音模组的被动播报方向：`0x81`
- 天气类扩展播报方向：`0x82`
- 品牌查询/应答链路还包含 `0x83`

### 3.2 上电、品牌与唤醒入口

- 模组上电后，需要主动向主控查询品牌信息。
- 文档给出的当前逻辑是：
  - 模组上电后立即每 `200ms` 发送一次品牌查询指令 `0x0102`
  - 收到品牌应答后，立即确定当前品牌对应的唤醒入口并播报欢迎语
  - 若 `1s` 内未收到任何品牌信息，则默认按“好太太”处理，启用 `小好小好`
- 当前品牌与唤醒入口：

| 品牌 | 唤醒词 | 模组发送协议 | 品牌应答 |
| --- | --- | --- | --- |
| 好太太 | 小好小好 | `0x0001` | `0x0020` |
| 科徕尼 | 小科小科 | `0x0002` | `0x0021` |
| 晒客 | 小晒小晒 | `0x0003` | `0x0022` |

- `小优小优` 在当前表内已标记“取消”，不纳入当前有效集。

### 3.3 休眠与唤醒

- 常规休眠时长：`25 秒`
- 休眠提示语：`主人，我休息了`
- 唤醒应答语存在多条候选，如 `在呢`、`诶`、`好的`
- 需求重点不是固定某一条唤醒应答，而是：
  - 能被品牌对应唤醒词唤醒
  - 唤醒后能进入命令识别状态
  - 超时后能回到休眠等待状态

## 4. 全局配置与掉电保存

### 4.1 基础配置

- 音量等级：`1~5 档`
- 默认音量：`3 档`
- 发音人：`叶子（情感）`

### 4.2 掉电保存项

`固件逻辑补充` 当前明确要求以下内容掉电保存：

- 音量
- 语音控制功能开/关状态
- 播报开/关状态

### 4.3 恢复出厂设置

收到 `0x006C` 时，需要执行恢复出厂设置，至少恢复：

- 默认音量 = `3`
- 播报开关 = `开启`
- 语音功能 = `开启`

## 5. 全局状态机逻辑

### 5.1 播报开关 / 静音逻辑

当前有效逻辑如下：

- 识别到 `退出静音` 时：
  - 发送协议 `0x0045`
  - 本地状态切换为“开启播报”
- 识别到 `打开静音` 时：
  - 发送协议 `0x0046`
  - 本地状态切换为“关闭播报”
- 语音识别到“开/关播报”时，不需要等待 MCU 回包后才更新本地状态，应主动生效。

MCU 回传对播报开关也有强制控制：

- 收到 `0x0069`
  - 不管当前播报开关状态如何，都切换为“开启播报”
  - 必须播报：`语音播报已开启`
- 收到 `0x0082`
  - 切换为“关闭播报”
  - 不做播报

### 5.2 语音功能开关逻辑

当前有效逻辑由 `固件逻辑补充` + `语音关闭逻辑.png` 共同确定：

- `关闭语音` 后，不是不再识别任何声音，而是进入“受限交互状态”
- 在该状态下：
  - 仍能识别唤醒词
  - 识别到唤醒词时，主动响应一声 `咚`
  - 同时刷新 `10 秒` 倒计时
  - 若仍处在整轮 `25 秒` 休眠计时范围内，则继续受限等待
- 在该受限窗口内：
  - 若识别到的不是 `语音功能打开`（包括再次说唤醒词、说普通命令词）
    - 主动提示：`语音功能已关闭，如需使用请对我说语音功能打开`
    - 不下发该业务指令给 MCU
  - 若识别到 `语音功能打开`
    - 退出受限状态
    - 恢复语音控制功能
    - 向 MCU 下发“语音功能打开”协议

补充规则：

- 重新打开语音功能时，如果当前“播报功能”为关闭状态，则需要同时恢复播报，并进行提示播报。
- 收到被动协议 `0x0012` 时，需要执行“关闭语音功能”操作。

### 5.3 被动播报与异常提示

- 默认规则仍是“以 MCU 回传协议为准播报”。
- 本轮迭代新增一条特殊要求：
  - 新增 `0x008C`
  - 含义：`设置失败`
  - 备注：文档写法是“新增被动播报：0x008C，设置失败，未播报”，当前更合理的理解是：
    - 新增一个失败状态协议
    - 语音侧不做实际播报
    - 可作为状态或日志事件处理

## 8. 当前整理出的待确认项

以下问题不影响先做需求清单，但在进入正式测试点拆分前最好确认：

1. `开机` 在 Excel 中存在命令词和被动播报 `0x0023`，但“发送协议”单元格为空，需要确认最终下发码。
2. `开启语音 / 关闭语音` 两行的被动播报文本与动作语义明显反了：
   - `开启语音` 行写成了“语音功能已关闭”
   - `关闭语音` 行写成了“语音功能已开启”
   当前清单按“命令词语义 + 语音关闭流程图”理解，但协议回包口径仍需确认。
3. `最大音量 / 调小音量` 两行的被动播报内容也存在错位痕迹：
   - `最大音量` 行挂了“音量已减小 / 0x0042”
   - `调小音量` 行挂了“音量已增至最大 / 0x0043”
   当前清单按更合理语义做了归并，需要后续确认最终回包。
4. `0x008C` 的文字描述是“设置失败，未播报”，建议确认它到底是：
   - 仅作为状态码，不播报
   - 还是保留一个占位协议，后续可能补播报
5. `0x0069` 同时在不同方向上被用于：
   - MCU -> 模组：语音播报已开启
   - 模组 -> MCU：关闭阅读模式
   这在当前文档体系内不构成冲突，但测试时必须严格按“数据方向”区分。

## 9. 结论

- 以 `语音关闭逻辑.png` 明确“关语音后不是完全失活，而是进入受限状态机”
