# Trisolaris 全链路验证方法

本文件用于让新的 Codex / OpenClaw 实例复刻 Trisolaris 的核心能力：面对全新离线语音需求，或已有项目需求变更时，从需求理解到固件烧录、方案设计、用例执行、断言收敛和最终报告形成完整闭环。

## 1. 适用范围

- 新项目：用户给出新的需求目录、协议/词表/固件，需要从零建立验证方案和用例。
- 既有项目变更：用户替换或修改已有需求文档、协议表、词表、固件，需要识别差异并回归受影响链路。
- 多项目并行：同一 skill 内保留多个项目资产，按需求路径选择项目，不互相覆盖证据、方案和用例。

## 2. 每轮启动规则

1. 读取 `plan.md`；不存在则创建。
2. 明确本轮目标、项目目录、固件路径、端口映射、声卡 key。
3. 把本轮计划写回 `plan.md`，后续每个重要动作同步 done / doing / todo。
4. 先判断当前任务属于：新项目、已有项目变更、新固件复测、单项缺陷复测、仓库发布。
5. 若用户给出明确端口或声卡，以用户输入为准，不猜测 `/dev/ttyACM*` 顺序。

## 3. 需求理解流程

读取项目目录内的有效输入：

- 需求文档：`.md`、`.docx`、`.pdf` 等。
- 协议/词表：`.xlsx`、`.csv`、`.h`、`.json` 等。
- 迭代说明和补充说明。
- 流程图或状态机图片。
- 固件 `.bin`。
- 已有方案、用例、历史报告。

将需求拆成以下结构：

- 基础配置：默认音量、档位数、唤醒词、会话超时、播报开关、语音开关。
- 主动命令：语音输入、期望主动协议、是否需要唤醒、是否需要上下文。
- 被动协议：MCU 到语音模组协议、播报、状态改变、是否保存。
- 状态机：语音关闭/打开、播报开关、配网、学习/注册、异常恢复。
- 数值项：时间窗口、档位边界、默认档位、次数、阈值。
- 持久化：保存触发、保存完成标记、重启后的配置或行为。
- 负向/异常：未唤醒阻断、删除词条不得响应、无效协议、设置失败不播报。

## 4. 需求变更分析

若是已有项目变更，先做差异分析再改用例：

- 新增需求：新增方案、用例、runner 断言。
- 删除需求：删除正向断言，必要时新增“不得响应”负向用例。
- 协议变更：更新主动/被动协议、校验和、play id 断言。
- 词条变更：更新语音短语，保留同需求行的稳定别名。
- 默认值变更：必须 `config.clear -> reboot -> burn` 后重测默认值。
- 状态机变更：重写前置状态、触发动作、恢复路径和禁止协议断言。
- 固件变更：保留同一需求基线，烧录对应固件后重跑受影响项和核心回归项。

## 5. 验证方案设计

每条需求要映射到明确验证方法：

| 需求类型 | 验证方式 | 主要证据 |
| --- | --- | --- |
| 主动命令 | 播放唤醒词和命令词 | 协议口主动帧 |
| 被动协议 | 协议口注入 MCU 帧 | 日志接收、play id、配置刷新 |
| 主动到被动全链路 | 主动命令触发后夹具回 MCU 响应 | 主动帧、被动接收日志、play id |
| 数值项 | 先测固件真实值，再比对需求 | 时间戳、配置、边界序列 |
| 持久化 | 设置 -> 等保存 -> 重启 -> 读取配置/行为 | `save config success`、启动配置、协议行为 |
| 状态机 | 建立状态 -> 触发迁移 -> 验证新状态 | 协议顺序、禁止协议、play id、日志状态 |
| 负向/异常 | 构造非法或受限状态 | 目标协议不得出现且采集有效 |

方案文档必须写清楚：

- 测试口径和硬件映射。
- 固件和烧录方式。
- 各需求点的验证方法。
- 数值项的测量方法。
- 状态机前置和恢复策略。
- PASS / FAIL / BLOCKED / TODO 判定。
- 证据路径和报告结构。

## 6. 用例设计规则

每条用例至少包含：

- 用例 ID、标题、需求来源。
- 前置条件和基线恢复动作。
- 输入动作：语音、协议注入、上下电、等待窗口。
- 期望主动协议、被动协议、日志、play id。
- 禁止出现的协议、日志、play id。
- 采集窗口和重试策略。
- PASS / FAIL 断言。
- 证据目录。

用例分层执行：

1. 烧录和启动门禁。
2. 握手 / ready。
3. 基础主动命令。
4. 被动协议。
5. 主动 -> 被动响应 -> play id 全链路。
6. 数值专项。
7. 持久化。
8. 状态机专项。
9. 负向和异常。
10. 定向复测和缺陷回归。

## 7. 烧录规则

所有项目统一使用仓库烧录入口：

- Linux：`tools/burn_bundle/run_fan_burn.sh`
- Windows：`tools/burn_bundle/run_fan_burn.ps1`

统一进入烧录模式时序：

```text
断电 -> 进 boot -> 上电 -> 上电后保持 boot 6s -> 下 boot -> 启动 Uart_Burn_Tool
```

实现要求：

- 默认 `PreBurnWaitMs=6000`。
- 等待位置必须在 `uut-switch1.on` 之后、`uut-switch2.off` 之前。
- 不允许把等待放到下 boot 之后。
- 烧录成功不能只看工具 exit code，必须检查成功标记：
  - `CONNECT ROM AND DOWNLOAD RAM LOADER SUCCESS`
  - `SEND MD5 COMMAND WITH RAM SUCCESS`
  - `SEND END COMMAND SUCCESS`
- 验证默认值前，如设备支持完整清配置，执行 `config.clear -> reboot -> burn`。

## 8. 可测性门禁

烧录成功后先跑最小 gate，再进入全量：

- 启动日志可读。
- 固件版本正确。
- 协议口可收发。
- 控制口可上下电 / boot。
- 声卡播放成功。
- 唤醒词可识别。
- 基础命令可下发。
- 如果项目依赖 MCU 握手，品牌查询和心跳必须闭合。
- 不存在反复重启、算法初始化失败、`MCU is not ready!` 持续阻塞等不可测状态。

Gate 不通过时不得进入正式结论。先修环境、握手、端口、烧录或可测性问题。

## 9. 断言规则

证据优先级：

1. 协议结论看协议 UART 原始帧。
2. 日志 `send msg::` 只作辅助。
3. 识别、播报、保存、启动配置、超时看日志 UART。
4. 播报以 play id / `play start` / `play stop` 为证据，不用人耳判断。
5. 持久化必须先等保存完成再断电或重启。
6. 负向用例必须确认采集有效，不能因为空采集直接 PASS。

固定断言模式：

- 唤醒超时：测响应结束或 `0x0001/Wakeup` 到 `TIME_OUT/MODE=0` 的真实差值，再比对需求。
- 默认音量：清配置烧录后，从默认位向一侧探边界，再双边探总档位，按边界步数推默认档位。
- 音量档位：必须让 MCU 回包触发模块本地音量改变后，再读配置或播放器音量。
- 主动本地副作用：如果主动协议只是上报 MCU，则必须补 MCU 被动确认后再断言模块状态变化。
- 语音关闭：主动 `0x0016` 后需回被动 `0x0012`，再断言受限态。
- TTS 短语：单个短语稳定打到邻近意图时，用同需求行官方别名探测；若别名通过，不保留功能 FAIL。

## 10. FAIL 收敛决策树

出现 raw FAIL 后按顺序排查：

1. 固件是否烧录成功。
2. 固件版本是否正确。
3. 默认值是否经过 `config.clear -> reboot -> burn`。
4. 串口和声卡映射是否正确。
5. 音频是否播放成功。
6. 协议握手是否 ready。
7. 采集窗口是否足够。
8. 是否有跨用例状态污染。
9. 是否缺少保存闭环。
10. 是否缺少 MCU 被动回包。
11. 断言是否写错。
12. 语音短语/TTS 是否导致误识别。
13. 需求是否矛盾、缺失或不可验证。
14. 固件是否真实不满足需求。

只有第 13 或第 14 类问题可保留为最终 FAIL。第 1 到第 12 类必须修验证逻辑并重跑。

## 11. 最终报告要求

最终报告必须包含：

- 项目名、需求目录、固件路径。
- 烧录结论和成功标记。
- 设备映射和测试口径。
- 用例总数、PASS、FAIL、BLOCKED、TODO。
- 分模块统计。
- 每条 FAIL 的需求期望、实测现象、证据路径、归因。
- 已收敛掉的测试方案/断言问题。
- 后续待办。

最终 FAIL 只能归类为：

- 固件问题。
- 需求问题。

不得把测试方案、用例设计、断言、采集、环境、状态污染、TTS 短语选择等问题留在最终 FAIL 中。

## 12. 发布和复用

别人复刻 skill 能力时，优先给完整仓库：

```text
SKILL.md
README.md
AGENTS.md
references/
tools/
deliverables/*/plan/
deliverables/*/cases/
项目需求/
项目需求/CSK5062小度风扇需求/
```

不需要发布运行产物：

```text
result/
audio_cache/
deliverables/*/reports/
tools/burn_bundle/*/app.bin
tools/burn_bundle/*/burn.log
tools/burn_bundle/*/burn_tool.log
__pycache__/
.venv/
```

发布前检查：

1. 更新 `plan.md`。
2. 执行相关脚本语法检查。
3. 检查 `.gitignore` 避免运行产物入库。
4. `git status --short` 确认提交范围。
5. commit 并 push 到远端。
