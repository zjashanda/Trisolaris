# Trisolaris 模块化验证池工作流

## 目标

把 skill 从“参考旧项目文档生成方案”改成“输入需求 + 固件后，按模块化验证池匹配逻辑，生成当前项目专属方案、用例、断言和执行链路”。

## 核心原则

1. 旧项目 deliverables 是案例，不是新项目默认规则。
2. 新项目必须先读 `references/validation-pool/INDEX.md` 和 `schema.md`。
3. 同一功能点允许多个项目变体；必须根据当前需求选择变体。
4. 需求未说明的关键维度，不得套用旧项目默认值。
5. 新逻辑经执行收敛后，抽象回灌到验证池；项目码值和项目结论留在项目 deliverables。

## 标准流程

### 1. 输入归档

- 读取需求文档、协议表、词表、tone/play id 表、流程图、固件和用户最新澄清。
- 把本轮输入复制或引用到项目需求目录。
- 建立 `source_inventory`，记录每个结论来源。

### 2. 功能点建模

将需求拆成统一结构：

```text
功能点 = 意图 + 触发 + 前置状态 + 期望输出 + 状态变化 + 数值约束 + 证据来源
```

同时为每个功能点打标签，例如：

- `wake-session`
- `active-protocol`
- `passive-protocol`
- `active-passive-closed-loop`
- `broadcast-play-id`
- `volume-level`
- `persistence-config`
- `state-machine-settings`
- `voice-switch-state-machine`
- `voice-register`
- `multi-wake`
- `negative-abnormal`
- `packaging-config-matrix`

### 3. 模块匹配

- 先用 `tools/pool/validation_pool.py classify` 做关键词候选匹配。
- 再人工/agent 读取命中模块，确认变体。
- 如果命中多个互斥变体，标为 `AMBIGUOUS`，生成最小探测或澄清项。

### 4. 生成项目方案

每个功能点都落成以下映射：

| 需求点 | 模块 | 变体 | 用例 | 断言 | 执行器 | 证据 |
| --- | --- | --- | --- | --- | --- | --- |

方案必须包含：

- 正向用例；
- 必要反例；
- 边界/数值用例；
- 状态恢复用例；
- 人工/TODO/BLOCKED 项。

### 5. 执行前门禁

- 烧录时序正确；
- 日志/协议/控制/声卡可用；
- 控制口必须实测能触发断电、上电和 boot 时序；如果用户给出的 `/dev/ttyACM*` 与当前枚举不一致，先记录端口探测结论，再进入验证；
- 设备健康检查通过；
- 最小唤醒/命令 smoke 通过；
- MCU ready 或协议仿真 ready 闭合。

### 6. 执行和收敛

- 串口采集不能并发占用。
- 正式全集执行必须优先走通用入口 `tools/suite/run_formal_suite.py`；该入口负责项目识别、模块匹配、烧录/gate、分组隔离、补充专项和最终聚合，对外只输出一次全集结果。
- 通用入口不能写死某个项目的协议码值或断言；项目差异必须放在 `references/project-profiles/*.json` 和对应 adapter / 项目 runner 中。
- 内部可以按模块分阶段执行，但最终报告必须是正式用例全集视角，不能让用户手工拼接“主 fullflow + 补充用例 + 聚合”。
- raw FAIL 先按 `fault-convergence.md` 排查。
- 对被动协议或主动-被动闭环类 FAIL，先确认协议注入发生在设备 ready 之后，且没有与心跳/周期帧粘连；必要时延后注入或延长尾部采集窗口后复测。
- 测试逻辑问题必须修复后重跑。
- 最终 FAIL 只能是固件问题或需求问题。

### 7. 回灌验证池

当出现以下任一情况，必须回灌：

- 新功能类型；
- 已有功能的新变体；
- 新断言证据；
- 新门禁或环境规则；
- 新 raw FAIL 收敛步骤；
- 新的反例/异常覆盖方式。

回灌位置：

- 通用逻辑 -> `references/validation-pool/*.md`
- 项目码值/项目结论 -> `deliverables/<project>/...`
- runner 通用能力 -> `tools/` 可参数化脚本
- 项目专用 runner -> `tools/debug/run_<project>_*.py`

## 同功能不同项目的处理

同名功能必须先拆“意图”和“实现变体”：

| 功能意图 | 可能变体 | 处理方式 |
| --- | --- | --- |
| 关闭语音 | 完全不响应 / 仍可唤醒但业务受限 / 当前会话和超时后不同窗口 | 读取需求状态机，不套用好太太规则 |
| 音量默认值 | 固件默认 / 打包默认 / 配置保存后默认 / raw 编码与档位不一致 | 优先用边界探测确认用户档位 |
| 播报 | 必须播报 / 不播报 / 播报关闭后静默 / 提示音替代语音 | 先建前置状态，再限定目标动作播报窗口 |
| 设置模式 | 设置窗口内普通词阻断 / 普通词可执行但窗口保持 | 按需求和探测结果选择变体 |
| 多唤醒 | 默认词始终有效 / 非当前词失效 / 多词均可唤醒 | 每种模式单独结论 |
| 平台打包 | 运行态固件验证 / 配置断言 / 控制变量包 | 不把配置问题误判为固件运行问题 |

## 最小交付物

新项目完成后至少输出：

- `当前有效需求清单.md`
- `模块化验证池匹配结果.md`
- `测试方案.md`
- `测试用例.md` 或 xlsx
- `执行报告.md`
- `FAIL 收敛结论.md`
- 验证池回灌变更
