# 平台打包、配置矩阵和控制变量复测

## 适用需求特征

- 需求不是单个固件，而是需要按平台参数打包生成多个固件并验证。
- 需要覆盖数值边界、布尔开关、枚举项、依赖参数和配置断言。

## 变体维度

- 固件固定验证 / 平台打包验证。
- 单包基线 / 多包边界矩阵 / 控制变量隔离包。
- 可运行验证 / 只读观察 / 不支持裁剪。

## 需求解析字段

- 产品、模块、语言、版本、发音人、配置项、值域、依赖、支持状态。

## 验证方案模板

1. 查询平台支持矩阵，确认产品/模块/语言/版本/voice/vcn 支持。
2. 裁剪本产品不支持功能，不生成失败结论。
3. 设计最少包数覆盖最多参数：中值包、左边界包、右边界包、依赖开启包、依赖关闭包。
4. 每包先做配置断言，再烧录和运行态验证。
5. 组合包失败后还原其他参数，只保留问题点和最小依赖复测。

## 用例模板

- `PKG-MID-STABLE-001`
- `PKG-LEFT-BOUNDARY-001`
- `PKG-RIGHT-BOUNDARY-001`
- `PKG-DEPENDENCY-ON-001`
- `PKG-DEPENDENCY-OFF-001`
- `PKG-ISOLATION-001`

## 断言与证据

- 不支持功能点透明披露，不计 FAIL。
- 配置未入包是平台/模板映射问题，不是运行态功能问题。
- `vcn` 必须与语言匹配，不能擅自替换用户配置。
- 组合失败必须用控制变量证明根因。

## 执行器映射

- mars-belt：`scripts/mars_belt.py`、`listenai_profile_suite.py`、weekly runner。

## 回灌规则

- 新平台参数需记录值域、配置断言路径、运行态证明方式和依赖项。
