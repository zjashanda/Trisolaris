# 多文件需求解析与功能点建模

## 适用需求特征

- 输入包含多份需求文档、词表 Excel、协议表、tone/play id 表、流程图、补充说明和固件。
- 需求是老项目变更或新项目首次接入。

## 变体维度

- 输入形式：自然语言文档 / Excel 词表 / tone.h / JSON 配置 / 图片流程图。
- 需求类型：功能命令 / 配置参数 / 状态机 / 平台打包参数 / 删除项。
- 结论口径：自动化 / 人工确认 / TODO / BLOCKED。

## 需求解析字段

- 项目名、芯片、固件、语言、版本、端口、声卡。
- 默认状态：唤醒词、音量、模式、保存开关、注册开关。
- 动作：语音输入、主动协议、被动协议、播报、配置刷新。
- 状态：前置状态、目标状态、互斥状态、恢复路径。
- 数值：时长、档位、次数、阈值、边界。
- 删除/无效项：旧词条、取消范围、人工项。

## 验证方案模板

1. 建立 `当前有效需求清单`，先排除删除项和未定义项。
2. 给每个功能点打标签并匹配验证池模块。
3. 对同名功能做变体选择，不套用旧项目默认逻辑。
4. 给每个功能点生成 `需求 -> 模块 -> 变体 -> 用例 -> 断言 -> runner` 映射。
5. 对歧义项生成探测用例或 TODO，不直接判 FAIL。

## 用例模板

- `REQ-PARSE-001`：需求文件清单和输入完整性检查。
- `REQ-MAP-001`：功能点到验证池模块映射。
- `REQ-DEL-001`：删除项/无效项反例策略。
- `REQ-AMB-001`：歧义协议或文案最小探测。

## 断言与证据

- 项目需求清单必须能追溯到原始文档。
- 每个自动化用例必须绑定需求点和模块。
- 无定义或人工项不能伪装成 PASS 或固件 FAIL。

## 执行器映射

- mars-moon：`scripts/mars_moon_pipeline.py prepare/build` 生成 normalized spec 和 cases。
- Trisolaris：`tools/pool/validation_pool.py classify` 生成模块匹配文档。
- 好太太/小度：当前项目专用 runner 读取项目 deliverables 中的用例资产。

## 回灌规则

- 新需求只要产生新状态机、新断言模式或新证据规则，必须抽象到对应模块。
- 项目专属协议码、play id 和固件结论只留在项目 deliverables。
