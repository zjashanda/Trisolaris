# 模块化验证池文档规范

## 目标

验证池只沉淀可迁移的验证逻辑，不沉淀某个项目的最终结论。新项目输入需求后，先匹配验证池模块，再生成当前项目自己的方案、用例、runner 映射和断言。

## 模块边界

每个模块描述一种可复用的功能意图，例如“唤醒会话”“音量档位”“语音注册”。同一功能意图允许存在多个项目变体；不得把某项目实现当作唯一正确逻辑。

## 必填结构

每个 `*.md` 模块必须包含：

1. YAML front matter
   - `module_id`：唯一模块 ID
   - `title`：中文标题
   - `tags`：用于需求匹配的关键词
   - `source_projects`：已抽象来源项目
2. `## 适用需求特征`
3. `## 变体维度`
4. `## 需求解析字段`
5. `## 验证方案模板`
6. `## 用例模板`
7. `## 断言与证据`
8. `## 执行器映射`
9. `## 回灌规则`

## 变体处理规则

同一功能点在不同项目里可能完全不同，必须按变体处理：

- 先识别功能意图，不直接选断言。
- 再根据需求文本、协议表、流程图、配置项和用户澄清选择变体。
- 如果同一需求命中多个变体，标记为 `AMBIGUOUS`，必须通过最小探测或用户澄清确认。
- 如果需求没有说明某个关键维度，不得套用旧项目默认值；只能生成 TODO/BLOCKED 或探测型用例。
- 项目特例必须写入项目 deliverables；只有抽象成通用变体后才能回灌本池。

## 输出要求

模块匹配后应生成项目级文档：

- 功能点清单；
- 匹配模块和变体；
- 采用/不采用的逻辑说明；
- 当前项目用例；
- 断言和证据路径；
- 未覆盖或需人工澄清的项目。

## 最终 FAIL 规则

最终 FAIL 只能是：

1. 固件不满足当前需求；
2. 当前需求自相矛盾或错误。

验证方案、断言、采集、环境、状态污染、TTS 短语、MCU 回包、打包配置、串口/声卡等问题必须先收敛后复跑。
