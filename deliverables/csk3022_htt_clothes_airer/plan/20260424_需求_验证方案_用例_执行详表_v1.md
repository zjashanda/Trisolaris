# 好太太晾衣机 需求--验证方案--用例--执行详表 v1

更新时间：2026-04-24

## 1. 文档目的和当前口径

这份文档把当前好太太晾衣机需求按“需求解析 -> 验证方案 -> 用例/执行逻辑 -> 断言 -> 实测数据 -> 结果”串起来，便于审核我根据需求生成的测试内容是否符合要求。

当前结论按“当前有效功能需求唯一点”统计，不重复累计重跑用例：

| 统计口径 | 数量 | 说明 |
| --- | ---: | --- |
| 当前有效需求点 | 88 | 不含迭代明确删除项；包含握手/会话/数值/功能/配置/play id |
| PASS | 62 | 已有可复现执行证据，含主动协议、被动回包、play id、数值/持久化等 |
| FAIL | 24 | 3 个状态机/异常/计时类 + 21 个 active-only 功能点 |
| 不计稳定结论 | 2 | `MTR-02 开机`主动协议未定义；`LGT-04 调暗`台架/ASR flaky |
| 删除/非范围项 | 9 个意图 + 若干已取消协议 | 不作为 PASS 目标，只能做反例回归 |

重要修正口径：

- `打开微醺模式`不是独立功能点，是`打开彩虹模式`的别名/泛化说法，二者同属 `LGT-27 -> 0x0036`。
- `关闭阅读模式 -> 0x0069`是当前有效需求，已补测，结果 FAIL。
- active-only 唯一功能点仍是 21 个，但命令词执行条数是 22 条，因为 `彩虹/微醺`两种说法都要覆盖。
- 数值类不按需求值反推；所有数值结论都先从日志或运行边界测出实际值，再与需求值比对。

## 2. 需求来源和解析规则

| 来源 | 用法 | 解析规则 |

| `项目需求/好太太晾衣机/当前有效需求清单_20260422.md` | 已整理的有效需求清单 | 作为当前矩阵的基础口径，但本文件已补充 2026-04-24 的 active-only 修正 |

解析时按以下规则处理：

1. 先识别功能是否仍在当前有效范围；迭代明确删除的意图不作为正例 PASS 目标。
2. 主动命令以 `CSK -> MCU` 的 `0x7F` 帧为核心证据；心跳/握手帧只证明链路，不计业务通过。
3. 被动播报以 `MCU -> CSK` 的 `0x81` 注入帧、日志接收、play id、play 次数和配置状态为核心证据。
4. active-only 功能如果没有固定 MCU 回包，不能强制断言播报，只断言“唤醒 -> 识别 -> 目标主动协议”。
5. 数值项必须先测实际值：例如超时取 `TIME_OUT + MODE=0`，音量取实际边界序列和配置值，不用需求值倒推。
6. 状态机项要同时看正例、反例和恢复链路：关闭语音后能唤醒但普通业务必须被阻断；重新开启后业务必须恢复。

## 4. 通用验证方案库

| 方法ID | 适用需求 | 执行逻辑 | 断言 |
| --- | --- | --- | --- |
| `M-HS-01` | 上电品牌/心跳 ready | 上电后监听 `COM36/COM38`，夹具回品牌和心跳 | 出现 `0x0102`、`0x5A5A/A5A5`闭环；无持续 `MCU is not ready!` |
| `M-ACT-01` | 主动语音命令 | 播放`小好小好`，再播放命令词，抓 `COM36` 主动帧 | 主动协议按顺序包含 `[0x0001, 目标业务协议]` |
| `M-ACT-02` | 未唤醒反例 | 不播放唤醒词，直接播放业务命令 | 不得出现目标业务主动协议 |
| `M-FULL-01` | 有固定被动响应的全链路 | 主动命令后由夹具注入对应 `0x81` 回包 | 主动协议正确；被动帧被接收；响应 play id 与预期一致 |
| `M-PASSIVE-01` | 纯被动播报/配置 | 不播放语音，直接在 `COM36` 注入 `0x81` 帧 | 日志收到帧；play id/次数/配置状态符合需求 |
| `M-TIME-01` | 普通会话超时 | 以唤醒协议/识别日志/唤醒响应为起点，等 `TIME_OUT + MODE=0` | 实测间隔接近 25s；不能用等待窗口反推 |
| `M-TIME-02` | 命令后超时 | 以命令识别/发送业务协议/命令播报结束为起点，等 `TIME_OUT + MODE=0` | 与纯唤醒超时时间一致，误差不超过 1.5s |
| `M-VOL-01` | 音量默认/档位/边界 | 先恢复出厂，再用增大/减小音量向上下边界刺探 | 从实际 `mini player set vol` 序列算档位和默认档，再与需求比对 |
| `M-PERSIST-01` | 掉电保存 | 设置非默认状态，等待保存日志，重启后读取实际配置/行为 | 重启后实际配置保持，且功能表现与配置一致 |
| `M-STAT-VOICE` | 语音开关状态机 | 主动/被动关闭语音，验证受限态，再验证重新打开和业务恢复 | 关闭后阻断普通业务；开启后必须恢复普通业务 |
| `M-ABN-01` | 异常协议 | 注入异常/设置失败协议 | 不得错误播报或误切状态 |

## 10. 结论

- 当前按“有效功能需求点”口径，矩阵已经把需求、验证方案、用例、断言和执行结果一一对应到 88 点：`PASS 62 / FAIL 24 / 未定义或 flaky 2`。
- 当前稳定 FAIL 不是用例方案或断言问题：`G-07`、`CFG-12`、`G-08`和 active-only 21 项都经过复测，现象稳定。
- 当前方案已修正数值校验思路：唤醒超时、受限窗口和音量档位/默认值均先获取实际值，再与需求比对。
- 需要你重点审核的边界是第 9 节：是否把 Excel `协议播报` sheet 的全部纯被动诊断状态码也纳入正式需求。如果纳入，需要新增纯被动播报全量 sweep；如果不纳入，当前 88 点功能矩阵就是当前有效需求执行口径。
