高分模块和本项目采用变体如下：

| 模块 | 当前项目变体 | 采用原因 | 对应用例/runner |
| --- | --- | --- | --- |
| `requirement-ingestion` | 多文件需求 + Excel/协议/流程图 + 用户澄清 | 需求中既有原始词表，又有迭代澄清和流程图 | 当前有效需求清单、正式用例 |

| `active-protocol` | 语音识别后主动协议主断言 | 多数命令以 CSK -> MCU 主动帧为核心输出 | formal、sweep、active-only |
| `passive-protocol` | MCU 被动协议注入 | 需求含强制播报、状态设置、恢复出厂 | formal、follow-up |
| `active-passive-closed-loop` | 主动上报后 MCU 回包改变本地状态 | 语音开关、音量、播报开关等存在本地副作用 | formal、follow-up、state-machine |
| `broadcast-play-id` | play id 主断言，限定目标动作窗口 | 需求有大量播报和设置失败提示 | sweep、follow-up |
| `volume-level` | 单边 + 双边边界探测 | raw volume 编码不等于用户档位 | numeric |
| `persistence-config` | 等保存闭环后重启观察 | 音量/播报/语音开关掉电保存 | follow-up |
| `voice-switch-state-machine` | 好太太受限态变体：当前会话 25s、超时后 10s、打开语音恢复 | 用户已澄清 G-07 语音关闭逻辑 | state-machine |
| `negative-abnormal` | 未唤醒、受限态、播报关闭、超时后阻断 | 当前 96 条包含功能级反例 | formal、state-machine |
| `fault-convergence` | raw FAIL 先修断言/夹具/采集，再重跑 | 本项目已多次出现断言窗口、回包、TTS 别名问题 | 全套件 |

## 3. 不采用或低分模块

| `packaging-config-matrix` | 当前输入为固定固件，不是平台多包打包验证 |

## 4. 由池子约束出的关键断言

- 不能把旧项目语音关闭逻辑套到好太太；必须按当前需求和探测选择变体。
- 主动关闭语音采用 `0x0016 -> 0x0036`，主动打开语音采用 `0x0017 -> 0x0037`。
- 被动 `0x0012` 是 MCU 直接关闭语音入口。
- `0x008C` 按“设置失败”播报，期望 `play id 127`。
- 暖光类 TTS 短语采用同需求行稳定别名：`调暖一点`、`调到最暖`。
- 默认音量和总档位采用探测法，不用 raw 编码直接判。
- 负例必须证明采集有效，不接受空采集 PASS。
