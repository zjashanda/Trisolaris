# CSK5062小度风扇首批探索性测试用例

## 1. 说明

这份用例不是一次性铺满全量，而是先选最容易暴露方向性问题的 6 个点：

1. 当前固件身份确认
2. 烧录后版本确认
3. 关播报/开播报
4. 关语音/开语音
5. 切换唤醒词
6. 学习结果掉电保存

这样做的目的有两个：

- 先确认设备是不是已经跑在正确固件上
- 在功能覆盖上优先打中最核心的状态切换与持久化闭环

当前执行状态说明：

- `PASS`：已实测通过
- `FAIL`：已实测失败
- `BLOCKED`：当前被环境、烧录或端口配对阻塞，暂时无法进入真实功能验证
- `TODO`：已写好，待条件满足后执行

当前执行辅助约定：

- 本地合成语料统一缓存到 `audio_cache/tts/`
- 语料清单统一维护在 `audio_cache/manifest.json`
- 验证辅助脚本：`tools/audio/fan_validation_helper.py`
- 连续播语料时，必须等上一条响应播报结束后再播下一条；当前首批实测固定采用 `4.5s` 间隔作为保守等待窗口

### TC_ENV_001 当前固件身份确认

- 功能点：确认当前板子是否已经跑在 `CSK5062小度风扇` 目标固件上。
- 前置条件：
  - 串口 shell 可进入
  - 当前使用 `COM38 @ 115200`
- 步骤：
  1. 进入 `COM38` shell。
  2. 执行 `factory version`。
  3. 执行 `reboot`，抓取启动日志。
  4. 比对运行版本与目标目录 `fw-csk5062_xiaodu_fan-v1.0.0`。
- 预期结果：
  - 运行版本应明确对应风扇项目 `V1.0.0`。
  - 不应仍显示窗帘类配置项，如 `curtain type`。
- 当前结果：`PASS`
- 当前实测：
  - 重启日志出现 `version         :       1.0.0`
  - 配置项已切换为 `voice / wakeup / regSave*`
- 结论：
  - 当前设备已烧成目标风扇固件。
  - 证据：`current_com38_version_check.txt`

### TC_REPORT_001 关闭播报后主动/被动播报都被抑制

- 功能点：验证 `关闭播报` 后，系统仍识别并执行功能，但所有播报都应被抑制；`开启播报` 后恢复。
- 前置条件：
  - 已烧入目标风扇固件
  - 可正常语音唤醒
  - 可观测播报与协议
- 步骤：
  1. 唤醒设备。
  2. 说 `关闭播报`。
  3. 观察该条关闭播报命令是否返回 `播报已关闭`。
  4. 再说 `打开电风扇`，检查协议/动作是否生效。
  5. 验证此时不再有主动播报。
  6. 发送一条被动播报协议，验证同样不播报。
  7. 说 `开启播报`。
  8. 再说 `关闭电风扇`，确认播报恢复。
- 预期结果：
  - 关闭播报命令本身有确认播报
  - 关闭后主动播报、唤醒播报、被动播报都被抑制
  - 协议和功能动作仍正常
  - 开启后恢复正常播报
- 当前结果：`PASS`
- 当前实测：
  - `小度小度 -> 关闭播报`：
    - 识别成功
    - 协议日志出现 `A5 FA 09 BB`
    - 当前这条关闭播报命令仍有 `play start` / `play id : 15`
  - 关闭播报后再执行 `小度小度 -> 打开电风扇`：
    - 仍能识别 `xiao du xiao du`
    - 仍能识别 `da kai dian feng shan`
    - 仍下发 `A5 FA 01 BB`、`A5 FA 04 BB`
    - 但后续日志不再出现 `play start` / `play id`
  - 再执行 `小度小度 -> 开启播报 -> 小度小度 -> 打开电风扇`：
    - `开启播报` 识别成功，协议日志出现 `A5 FA 08 BB`
    - 后续唤醒与开风扇重新出现 `play start` / `play id`
- 证据：
  - 关闭播报链路：`result/0417134932_report_off_probe/serial_utf8.txt`
  - 开启播报恢复链路：`result/0417135050_report_on_probe/serial_utf8.txt`
- 注意点：
  - 必须区分“关播报”与“关语音”，不能因为不播报就误判成不执行
  - 被动播报必须纳入验证，不能只测主动命令播报

### TC_VOICE_001 关闭语音后不再响应唤醒与命令

- 功能点：验证 `关闭语音` 通过语音命令关闭全部语音识别入口；`开语音` 通过协议串口 `COM36@9600` 发送 `A5 FB 0A CC` 恢复；两者都掉电保持。
- 前置条件：
  - 已烧入 `1.0.0` 风扇固件
  - `COM38` 为日志口、`COM36` 为协议口、`COM39` 为断电控制口
  - 当前设备处于语音已开启状态
- 步骤：
  1. 执行 `小度小度 -> 关闭语音`。
  2. 确认 `COM36` 抓到 `A5 FA 01 BB A5 FA 11 BB`，`COM38` 出现 `keyword:guan bi yu yin`、`refresh config ... voice=0 ...`、`save config success`。
  3. 再执行 `小度小度 -> 打开电风扇`，确认 `COM36` 无新增协议、`COM38` 仅出现 `voice switch is off`。
  4. 通过 `COM39` 硬断电重启，确认启动配置出现 `voice           :           0`。
  5. 重启后再次执行 `小度小度 -> 打开电风扇`，确认仍无新增协议、仍只出现 `voice switch is off`。
  6. 通过 `COM36@9600` 发送 `A5 FB 0A CC` 打开语音。
  7. 确认 `COM38` 出现 `receive msg:: A5 FB 0A CC`、`refresh config ... voice=1 ...`、`save config success`。
  8. 再执行 `小度小度 -> 打开电风扇`，确认 `COM36` 恢复抓到 `A5 FA 01 BB A5 FA 04 BB`。
  9. 再次硬断电重启，确认启动配置出现 `voice           :           1`。
  10. 重启后再次执行 `小度小度 -> 打开电风扇`，确认唤醒与命令仍正常恢复。
- 预期结果：
  - `关闭语音` 后唤醒词、风扇命令词等语音入口全部失效，且不再下发控制协议。
  - `关闭语音` 状态掉电保持。
  - 协议发送 `A5 FB 0A CC` 后可恢复唤醒和命令控制。
  - `开语音` 状态掉电保持。
- 当前结果：`PASS`
- 证据：
  - 关语音设置成功：`result/041717_voice_off_dual_current_retry/`
  - 关语音后阻断：`result/041717_voice_off_then_wake_open_blocked/`
  - 关语音掉电保持：`result/041717_voice_off_powercycle_check/`
  - 掉电后仍阻断：`result/041717_voice_off_after_powercycle_open_blocked/`
  - 协议开语音：`result/041717_voice_on_proto_persist_prep/`
  - 开语音后恢复：`result/041717_voice_on_after_proto_open_recheck/`
  - 开语音掉电保持：`result/041717_voice_on_powercycle_check/`
  - 掉电后仍恢复：`result/041717_voice_on_after_powercycle_open_ok/`
- 注意点：
  - `关闭语音` 走语音命令入口；`开语音` 当前按协议口恢复验证，不能用“说开启语音”代替本轮正式结论。
  - 协议有效性统一以 `COM36` 原始 hex 为准，`COM38` 只用于看识别、模式切换和保存日志。
  - `关闭语音` 后日志里的 `voice switch is off` 是拒识成功证据，不应误判为异常。
  - 做掉电保持前必须先看到 `save config success`，否则不能直接判保持失败。
